# 上下文工程技术方案

目标：目标是把“对话记忆、任务状态、工具结果、知识检索、长上下文压缩”统一纳入一个可控的上下文系统，避免 Agent 越跑越乱、越聊越贵、越长越慢。

导向：建立一套 上下文分层、动态装配、按需写入、可回放 的机制。

最终要解决 5 个问题：
- 记不住：对话一长就丢失关键信息。
- 塞太多：上下文过长，成本高、速度慢、容易跑偏。
- 不会挑：不知道当前任务该用哪些历史信息。
- 不可信：工具结果、知识库内容、模型臆测混在一起。
- 不可控：无法追踪模型为什么这么答。

## 架构
可以把上下文工程拆成 6 层：
- 层级	作用	典型内容
- 系统层	固定规则与行为约束	system prompt、输出格式、工具使用规则
- 任务层	当前目标与执行状态	用户目标、子任务、当前步骤、待办列表
- 对话层	最近交互内容	最近几轮 user/assistant 消息
- 记忆层	长期稳定信息	用户偏好、项目背景、历史结论
- 检索层	外部知识补充	RAG 文档、历史工单、代码仓库、工具结果
- 工作层	运行中中间态	计划、草稿、推理摘要、工具返回结果

核心原则是：
不同层的数据有不同的生命周期、信任等级和注入时机。
