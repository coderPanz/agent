以下是根据之前的讨论，整理的 **Agent 架构模式对比表**：

| 架构模式 | 核心思想 | 优点 | 局限 | 上下文工程视角 |
|---------|---------|------|------|----------------|
| **ReAct**（推理与行动） | 思考→行动→观察的循环，边推理边调用工具 | • 结构清晰，易调试<br>• 支持多步工具使用<br>• 上下文自然线性增长 | • 串行执行，延迟高<br>• 缺少长期规划，易局部循环<br>• 过度依赖模型推理能力 | 线性增长需严格滑动窗口+摘要；每N步自动压缩历史步骤防止窗口爆炸 |
| **Plan-and-Execute**（先规划再执行） | 规划器生成步骤列表，执行器逐步执行，可动态重规划 | • 有全局视野，适合复杂任务<br>• 用户可审核计划，安全性高<br>• 可支持并行步骤 | • 初始计划可能偏离实际<br>• 计划本身占用大量token<br>• 对规划模型要求高 | 计划存外部内存，执行器只读当前步骤；重规划时仅更新受影响分支，避免全量膨胀 |
| **多Agent协作**（Multi-Agent） | 多个拥有不同角色/工具的Agent协同，通过消息或共享黑板通信 | • 领域分离，单一Agent提示简洁<br>• 易扩展，新增角色不影响旧有<br>• 辩论模式可提高准确性 | • 通信开销大，上下文易爆炸<br>• 一致性与协调难控制<br>• 调试链路长 | 必须用标准上下文信封传递压缩后任务，全局黑板（Redis/DB）共享状态，严禁全量历史传递 |
| **自主循环**（AutoGPT风格） | 主控循环自主设定子目标、调用工具、存储记忆，直到完成最终目标 | • 可处理极度开放的长任务<br>• 长时间自主运行 | • 极易陷入死循环或无关操作<br>• 缺乏及时的外部反馈适应<br>• Token消耗巨大 | 核心为虚拟上下文管理（如 MemGPT），工作窗口暂存主逻辑，长期记忆持久化到外部，按需换页 |
| **状态机/工作流图**（State Machine / Graph） | 有向图明确定义状态、转换条件和节点（LLM/工具/代码），如LangGraph | • 最高级别的可控与可预测性<br>• 易于监控、错误恢复与人机协同<br>• 复杂分支/并行逻辑清晰 | • 灵活性受限，需提前设计图结构<br>• 不适用非结构化开放探索 | State对象即结构化上下文容器，可精细决定信息去留，天然支持节点级压缩与裁剪 |
| **反思与自我修正**（Reflection / Self-Refine） | Agent生成输出后自我审视批评，基于反馈迭代改进 | • 显著提升代码、写作、推理等质量<br>• 无需外部反馈即可自我提升 | • 迭代轮次不可控，延迟与成本增加<br>• 模型可能无法识别深层错误<br>• 可能过度反思导致更差 | 上下文翻倍或多于翻倍，需设定严格迭代上限+早停机制，早期冗长输出可压缩后再进入下一轮 |
| **强化学习 / 策略驱动**（RL-based） | 用强化学习训练策略模型，直接映射状态到动作，可脱离LLM推理 | • 推理延迟极低<br>• 可在特定环境优化至极高效率 | • 训练成本极高，泛化差<br>• 可解释性低<br>• 安全对齐复杂 | 推理时上下文固定且极简，训练阶段需大量离线轨迹数据，不依赖动态记忆管道 |

---

### 关键决策速查
- **追求可控、可审核** → 状态机/工作流图
- **复杂多步、可分解任务** → Plan-and-Execute
- **需要多领域专业协作** → 多Agent协作
- **简单到中等复杂度，快速实现** → ReAct
- **对质量有极致要求** → ReAct + 反思/自我修正
- **长期开放、自主运行** → 自主循环 + 虚拟上下文管理
- **极低延迟、固定环境优化** → 强化学习/策略驱动

实际通用Agent工程中，常采用 **图式主调度（状态机） + ReAct叶子节点 + 必要时自我修正** 的混合架构，并统一接入上下文管道，实现架构复杂度与token消耗的解耦。



下面用一个具体场景来演示 **“图式主调度 + ReAct 叶子节点 + 必要时自我修正”** 混合架构的执行流程。

---

## 场景描述

用户输入：  
> “帮我查一下明天从北京到上海的航班，挑一个上午出发最便宜的，然后用邮件把行程发给 `zhangsan@example.com`。”

---

## 主状态图设计（状态机总控）

```
[理解意图] → [搜索航班]（ReAct 叶节点）
                 ↓ 得到航班列表
            [筛选最佳航班]（纯代码逻辑）
                 ↓ 找到目标航班
            [撰写邮件草稿] → [自我修正]（反思循环）
                 ↓ 邮件内容确定
            [发送邮件]（工具调用）
                 ↓
            [最终回复]
```

各个状态转换由**主图状态机**控制，每个节点执行满后根据返回结果决定下一个节点。

---

## 详细执行步骤（带注释标明模式）

### 1. 状态：理解意图
- **模式**：轻量级 LLM 调用（非 ReAct，仅分类）
- **动作**：从用户输入提取意图 `book_flight_and_notify`，槽位 `{date: "2026-05-09", from: "北京", to: "上海", preference: "上午最便宜", recipient: "zhangsan@example.com"}`。
- **上下文中写入**：`intent` 和 `slots`。

---

### 2. 状态：搜索航班（ReAct 叶子节点）

进入一个独立的 **ReAct 子 Agent**，它只拥有一个工具 `search_flights`。主状态机会给它一个精简的上下文信封：
```json
{
  "task": "搜索航班",
  "parameters": {"date": "2026-05-09", "from": "北京", "to": "上海"},
  "output_schema": "航班列表JSON"
}
```

**ReAct 内部过程**（3 步）：

- **Thought 1**：`“需要调用 search_flights，参数为日期、出发地、目的地。”`
- **Action 1**：调用 `search_flights(date="2026-05-09", from="北京", to="上海")`
- **Observation 1**：返回 12 个航班（含航班号、起飞时间、价格等）。

- **Thought 2**：`“已拿到12个航班，数据完整，无需进一步筛选，可以原样返回给主控。”`
- **Action 2**：`finish(flight_list)`

> 上下文增加：原始 ReAct 历史（Thought/Action/Observation），执行完后由上下文管道压缩为 `flight_list` 摘要，只保留结构化的航班数据，传递给下一个状态。

---

### 3. 状态：筛选最佳航班（纯代码逻辑）

不是 LLM 节点，是**确定性函数**，直接操作上一步的 `flight_list`。

- 筛选规则：`departure_time < 12:00`，按 `price` 升序取第一条。
- 结果：找到航班 `CA1234，08:30-10:40，¥520`。
- 写入上下文：`selected_flight`。

如果筛选为空呢？主状态机检测到空结果后，会跳转到 **“澄清/扩大搜索”** 状态（可再接一个 ReAct 节点，让 Agent 询问用户是否改时间或降条件）。

---

### 4. 状态：撰写邮件草稿 + 自我修正（反思循环）

主状态机进入 **“撰写邮件”** 复合状态，其内部是一个**自我修正循环**：

#### 4.1 生成草稿（LLM 调用）
- 输入：`selected_flight`、收件人、用户偏好。
- 输出：邮件草稿 v1（含主题、正文）。

#### 4.2 反思（Critique）
- 模型对自己生成的草稿进行审查，依据约束条件：  
  `“检查：是否包含航班号、时间、价格？语气是否礼貌？有无错别字？是否标注了收件人？”`
- 输出批评：`“主题缺少明确航班信息，正文中时间未标注上午，建议补充。”`

#### 4.3 修正（Refine）
- 根据批评意见生成 v2 草稿：  
  `主题：5月9日北京→上海 CA1234 行程（08:30-10:40）`  
  `正文中添加“上午8:30出发，上午10:40抵达”。`

#### 4.4 再次反思（第2轮）
- 审查 v2，确认所有约束满足，输出 `“OK”`，循环终止。

> 上下文注意：反思循环中，只保留最新的 (v_k, critique_k) 对，旧版本可压缩或丢弃，以节省窗口。严格限制最大迭代次数为 3 次，防止死循环。

---

### 5. 状态：发送邮件（工具调用）

- 调用邮件工具 `send_email(to="zhangsan@example.com", subject=..., body=...)`。
- 返回结果：`success`。
- 上下文中记录发送状态。

---

### 6. 状态：最终回复

- 主状态机调用最后的回复模板，向用户总结完成情况：  
  `“已为您预订明天上午8:30的CA1234航班（¥520），行程已发送至 zhangsan@example.com。”`

---

## 架构角色归纳

| 组件 | 架构模式 | 职责 |
|------|----------|------|
| 整体主流程 | **状态机/图** | 控制节点转移、分支、异常处理，确保流程可审计 |
| 航班搜索 | **ReAct** | 多步思考与工具调用，灵活应对非确定性数据 |
| 航班筛选 | **确定性代码** | 规则明确，无需模型，0 成本 0 延迟 |
| 邮件撰写 | **自我修正** | 提升文本质量，确保准确性、合规性 |
| 邮件发送 | **工具调用** | 单一 API 调用，无推理 |

这种混合架构既保留了状态机的可控性和可调试性，又在需要推理的局部节点注入 ReAct 的灵活性，并在对质量敏感的输出环节加入自我修正，做到了**效率、质量、可控性**的平衡。
